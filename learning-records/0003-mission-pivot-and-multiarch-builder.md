# Mission expanded to multi-platform builds; created first multi-arch builder

The user arrived with their own notes on Buildx/BuildKit architecture (client/server split, layer caching, cache-miss propagation, Dockerfile instruction ordering for cache efficiency) and asked directly to create a local builder for multi-architecture builds. Since `MISSION.md` explicitly excluded multi-platform builds as out-of-scope, this was flagged and the user confirmed expanding the mission rather than treating it as a one-off tangent.

**Evidence**: Created a real `docker-container` builder (`multiarch`) on the user's machine and ran `docker buildx build --platform linux/amd64,linux/arm64` against `exercises/node-express` (their own Dockerfile from Lesson 0002). Both platforms built successfully; arm64 ran under QEMU emulation, visibly slower on the `RUN` steps than the native amd64 side. Discovered and corrected a docs-vs-reality gap: `docker buildx inspect` doesn't list `arm64` in the builder's platform list (only lists natively-supported platforms), but the build succeeded anyway via host-level `binfmt_misc` QEMU registration — the lesson documents this explicitly since the idealized docs example doesn't mention it.

**Implications**: The user's conceptual grasp of the build engine (Buildx vs BuildKit, caching) was already solid going in — this was knowledge, not the gap. The new skill acquired is operational: creating a `docker-container` builder and running a real multi-platform build. Next step in this track would be pushing a multi-arch manifest to a registry (currently out of scope per `MISSION.md`) or exploring `--driver-opt` and builder cache persistence (`buildx rm --keep-state`).
