Compose File Anatomy

Quick reference · top-level keys, per-service keys, and the app lifecycle

Top-level keys

A compose.yaml has a small set of top-level keys. Each declares a kind of thing the app is made of.

services:

The containers. Each child key is one service; its name doubles as the network hostname. Required.

networks:

Named virtual networks the services join. Omit it and Compose still makes a default network all services share.

volumes:

Named, persistent storage that outlives containers. Declared here, then mounted per service.

Common per-service keys

image:Use a published image (e.g. redis:alpine).
build:Build the image from a Dockerfile. Value is the build context (e.g. .).
ports:published (host) → target (in container). Short form: "5001:8080".
networks:Which declared networks this service joins.
volumes:source (named volume) → target (path in container).
environment:Env vars passed into the container.
depends_on:Start ordering only — does not wait for the dependency to be ready.
command:Override the process the container runs.

Service discovery (the key idea)

On a shared network, the service name is the hostname. A service named redis is reachable from other services at redis:6379. Never hard-code container IPs.

Lifecycle commands

Run from the directory containing compose.yaml. Modern Docker uses the docker compose subcommand (space, not the old docker-compose hyphen).

docker compose upBuild/pull, create, and start everything (attached to logs).
docker compose up -dSame, but detached (runs in the background).
docker compose psList this app's running services.
docker compose logs -f web-feFollow the logs of one service.
docker compose configPrint the fully-resolved file (great for catching YAML mistakes).
docker compose downStop and remove containers + networks. Add -v to also drop named volumes.
docker compose buildRebuild images without starting.
