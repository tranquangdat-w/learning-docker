Builder Drivers & Output Flags

Reference · where does a build's result actually go?

The rule

The docker driver always auto-loads to docker image ls. Every other driver never does — you must say --load or --push, every time, single or multi-platform.
Driver Runs where Auto-loads to docker image ls? Multi-platform builds?
docker (default) Inside the Docker Engine itself Yes, always No — single platform only
docker-container A separate BuildKit container No — needs --load or --push Yes
remote, kubernetes A remote/cluster BuildKit instance No — needs --load or --push Yes

--load and multi-platform: it depends on your image store

Docker Engine 29+/Docker Desktop can use one of two local image stores. Check yours:

docker info --format '{{.DriverStatus}}'
containerd image store

Shows io.containerd.snapshotter.v1 in the output above. --load works even for multi-platform builds — the result is stored as one OCI index containing every platform variant. Run docker image ls --tree to see them broken out per-platform.

classic image store

Older engines. --load only accepts a single platform — a multi-platform --load fails. Use --push to a registry instead; the registry assembles the manifest list, and nothing needs to live locally.

Quick decision guide

  • Just want to prove a multi-platform build works, don't need the image? No flag. It builds, stays in the build cache, and gets discarded (or reused as cache for your next build).
  • Want to run/inspect the image locally? Add --load. Fails on classic image store for multi-platform — check docker info first.
  • Want to ship it somewhere else / keep it permanently? Add --push with a registry-qualified -t (e.g. docker.io/<user>/name:tag). Needs docker login first.

Inspecting what you built

Where the image is Command to inspect it
Local, after --load docker image ls --tree
Registry, after --push docker buildx imagetools inspect <registry-ref>
Still only in build cache (no flag) docker buildx du --builder <name> (shows cache entries, not a runnable image)

docker buildx imagetools inspect only works against a registry reference — pointing it at a local-only tag like local/foo:bar fails with a misleading "pull access denied," because it assumes the name is a registry path.

Cleaning up build cache

docker buildx du --builder <name>      # see what's cached
docker buildx prune --builder <name>   # remove reclaimable cache (asks to confirm)
docker buildx prune --builder <name> -a -f   # remove everything, no confirm
