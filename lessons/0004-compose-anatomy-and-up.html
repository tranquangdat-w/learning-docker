Read a Compose File and Bring the App Up

Lesson 0004 · one file, two services, one command — and why the app finds Redis by name

The problem Compose solves

You already know how to build one image and run one container. But the app in ddd-book/multi-container is really two containers that must run together: a Flask web front end and a Redis data store. By hand that means two docker build/docker run commands, a network you create yourself, and a volume you wire up — every time, in the right order.

A Compose file replaces all of that. It declares the whole application — every service, network, and volume — in one compose.yaml. One command reads it and makes reality match the file. This is the same declarative idea as a Dockerfile, but one level up: a Dockerfile describes one image; a Compose file describes a running multi-service app.

The anatomy of compose.yaml

Here is the file you already have, trimmed to what matters:

networks:
  counter-net:
volumes:
  counter-vol:

services:
  web-fe:
    build: .
    command: python app/app.py
    ports:
      - target: 8080
        published: 5001
    networks:
      - counter-net
  redis:
    image: redis:alpine
    networks:
      - counter-net
    volumes:
      - type: volume
        source: counter-vol
        target: /data

Three top-level keys declare the pieces of the app. Read them as nouns — things that will exist once the app is up:

services:

The containers that make up the app. Each key under it (web-fe, redis) is one service. The name you choose here matters — hold that thought for the next section.

networks:

A private virtual network Compose creates for the app. Both services join counter-net, which is what lets them talk to each other while staying isolated from everything else on your machine.

volumes:

Named, persistent storage that outlives the container. counter-vol is mounted into Redis at /data, so the visit counter survives a restart.

Now the two services. The key difference between them ties straight back to what you already know about images:

web-fe → build: .

This service has no ready-made image, so Compose builds one from the Dockerfile in . (the build context you learned about earlier). This is your multi-stage/Dockerfile knowledge plugged straight into Compose.

redis → image: redis:alpine

This service pulls a published image from a registry instead of building. build: and image: are the two ways a service gets its image — build your own, or use someone else's.

ports: target 8080 / published 5001

target is the port inside the container (the Flask app listens on 8080). published is the port on your machine. So you build for 8080 but visit localhost:5001. Only web-fe publishes a port; Redis stays reachable only inside the network.

The aha: services find each other by name

Open app/app.py and look at line 6:

cache = redis.Redis(host='redis', port=6379)

The host is the literal string 'redis' — not an IP address, not localhost. That works because Compose runs an internal DNS on counter-net that resolves every service name to that service's container. The redis service is reachable at the hostname redis; if the service were named db, the code would say host='db'.

This is service discovery. The service name in compose.yaml is the network hostname. It's the single biggest reason Compose feels like magic — you never hard-code container IPs.

Bring it up

From inside ddd-book/multi-container, one command builds what needs building, pulls what needs pulling, creates the network and volume, and starts both services:

docker compose up

Watch the log: you'll see web-fe and redis start, interleaved. Then open http://localhost:5001 — every refresh increments the counter, which the Flask service is reading from Redis over the network you just declared. Stop it with Ctrl-C, then tear the whole app down cleanly:

docker compose down

Verify it yourself. Before you open the browser, run docker compose ps in another terminal to see both services listed, and docker compose logs redis to watch Redis alone. Reading the real output is how this workspace checks claims — don't take the log's word for it, look.

Check yourself

1. On the shared network, what does the Flask app use as the Redis hostname?
2. Which single command builds, creates, and starts every service?
3. With published: 5001 and target: 8080, which port do you open in your browser?
