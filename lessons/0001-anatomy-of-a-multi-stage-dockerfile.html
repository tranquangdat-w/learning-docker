Anatomy of a Multi-Stage Dockerfile

Lesson 0001 · the three structural moves you must recognize before you can write one

You already understand the why (smaller, safer images) and the build mechanics (build context, RUN in-container, COPY). This lesson compresses a multi-stage Dockerfile into three moves you can spot, name, and later write yourself.

Move 1 — Multiple FROM lines, each named

Every FROM starts a new, isolated stage. Name a stage with AS <name> so later stages can reference it. A builder stage is usually fat; the final stage is usually tiny.

FROM golang:1.23-alpine AS base

A heavy image with the compiler. Named base so others can reuse it.

Move 2 — COPY --from carries artifacts forward

The final stage does not rebuild. It starts from a minimal base image (or scratch) and copies only the finished binary out of the builder with COPY --from=<stage>. Anything not copied — compiler, source, caches — stays behind.

COPY --from=build-server /bin/server /bin/

"Take the binary from the build-server stage and drop it here."

Move 3 — The last stage is the shipped image

Docker builds every stage but only the last stage becomes the image you push. Earlier stages contribute only what a later stage explicitly copies. Use docker build --target <name> to stop at any named stage (handy for a test or builder stage in CI).

One-line mental model: fat builder → COPY --from the artifact → tiny final stage → that's the image.

Check yourself

1 · Which stage becomes the final shipped image?
2 · What does COPY --from=builder do?
3 · Where does a RUN instruction execute?
4 · What is the build context?
