Create a Local Builder for Multi-Architecture Builds

Lesson 0003 · your default builder can't do this — you'll build one that can

The problem

Run this on your machine:

docker buildx ls

You'll see something like:

NAME/NODE     DRIVER/ENDPOINT   STATUS    BUILDKIT   PLATFORMS
default*      docker
 \_ default    \_ default       running   v0.31.0    linux/amd64 (+3)

That default builder uses the docker driver — the built-in BuildKit instance that ships inside the Docker Engine itself. Per Docker's own docs, this driver "only supports single-platform builds and lacks the infrastructure for managing manifest lists required by multi-architecture images." The (+3) you see is just amd64 variants (v2, v3, 386) — not a second architecture.

Try to build for two architectures at once with it, and it won't work. You need a different driver.

The fix: a docker-container builder

The docker-container driver runs BuildKit inside its own dedicated container, separate from the Docker Engine. That separate container is what knows how to fan a build out across architectures and assemble the result into a manifest list — one image reference that points to several architecture-specific variants.

Create one:

docker buildx create --name multiarch --driver docker-container --bootstrap --use
--name multiarch

What you'll call this builder in every later command. Pick anything; docker buildx ls will show it by this name.

--driver docker-container

The one flag that actually matters here. Without it, buildx defaults to docker-container too when you omit the flag entirely — but naming it explicitly makes the choice visible, which matters while you're learning it.

--bootstrap

Starts the BuildKit container immediately. Without it, the builder is only registered — it doesn't actually launch until your first build.

--use

Makes multiarch the current default builder, so you don't need --builder multiarch on every command after this. (The lesson below still passes it explicitly, so each command is self-contained.)

Verify it

docker buildx ls
NAME/NODE        DRIVER/ENDPOINT   STATUS    BUILDKIT   PLATFORMS
multiarch*       docker-container
 \_ multiarch0    \_ ...           running   v0.30.0    linux/amd64 (+3), linux/386
default          docker
 \_ default       \_ default       running   v0.31.0    linux/amd64 (+3)
Notice: no arm64 in that platform list either. That list only shows what BuildKit can run natively. arm64 support comes from QEMU emulation, registered at the host kernel level (via binfmt_misc) — it doesn't show up in buildx inspect, but it's there and it works, as you're about to see.

The exercise: build your Node app for two architectures

You already wrote the Dockerfile for exercises/node-express/ in Lesson 0002. It's a pure-JavaScript app with no native dependencies — the safest possible thing to build across architectures, since nothing needs to be compiled differently per-arch.

Run:

docker buildx build --builder multiarch \
  --platform linux/amd64,linux/arm64 \
  -t local/node-express:multiarch \
  ./exercises/node-express

Watch the output closely. You'll see every stage run twice — once tagged [linux/amd64 ...], once [linux/arm64 ...]. The arm64 side is running under QEMU emulation: same instructions, translated CPU-instruction-by-instruction, which is why it's noticeably slower than the native amd64 side for anything CPU-bound (an npm ci or RUN compile step is where you'll feel it most).

You'll see a warning at the end: 
WARNING: No output specified with docker-container driver. Build result will
only remain in the build cache. To push result image into registry use --push
or to load image into docker use --load

This is expected, not an error. The docker-container driver builds in its own separate container, so it never lands in your local docker image ls automatically — unlike the docker driver, which always does, with no flag needed (see the reference table below). You have to say explicitly where the result goes: --load (into your local Docker) or --push (to a registry).

Correction if you're re-reading this lesson: an earlier version of this lesson claimed a multi-platform result "can't be --loaded." That's wrong on this machine. Whether --load works for multi-platform depends on which local image store your Docker Engine uses:
  • containerd image store (Docker Engine 29+/Docker Desktop default) — --load works for multi-platform. The result lands as one image entry containing every platform variant; run docker image ls --tree to see them broken out.
  • classic image store (older engines) — --load only accepts a single platform. A multi-platform --load fails outright; you'd need --push instead.
Check yours with docker info --format '{{.DriverStatus}}' — if it mentions io.containerd.snapshotter.v1, you're on containerd and --load just works. See reference/builder-drivers.html for the full table.

Solution recap

Reveal after you've run it yourself 
# one-time setup
docker buildx create --name multiarch --driver docker-container --bootstrap --use

# build only (proves it works, nothing kept locally)
docker buildx build --builder multiarch \
  --platform linux/amd64,linux/arm64 \
  -t local/node-express:multiarch \
  ./exercises/node-express

# build + keep it locally (needs containerd image store, see callout above)
docker buildx build --builder multiarch \
  --platform linux/amd64,linux/arm64 \
  -t local/node-express:multiarch --load \
  ./exercises/node-express

docker image ls --tree   # see both platform variants under one entry
Why can docker-container do this and docker can't? The docker driver is the Docker Engine's own built-in BuildKit — one instance, tied to the host's single architecture, with no manifest-list assembly logic. The docker-container driver spins up a separate BuildKit instance in its own container, purpose-built to run a matrix of platform builds in parallel (native or emulated) and stitch the results into one multi-arch reference. It's a different, more capable engine — not a configuration flag on the same one.
